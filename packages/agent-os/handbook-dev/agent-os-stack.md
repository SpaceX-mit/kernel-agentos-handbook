## 1. 概述

### 1.1 Agent OS 的完整定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Agent OS 完整定义                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   Agent OS  =  Agent Kernel  +  Agent Framework                             │
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │                                                                       │   │
│   │   Agent Kernel (agent-os):                                           │   │
│   │   ├── 定义: Agent OS 的 POSIX 标准接口                               │   │
│   │   ├── 职责: 提供基础抽象（进程/权限/配额/隔离/审计）                 │   │
│   │   ├── 性质: 与实现无关，任何 Kernel 都可以实现这些接口               │   │
│   │   │                                                                 │   │
│   │   │   spawn() / acap / chexec / ulimit-tok / handoff / vault...    │   │
│   │   │                                                                 │   │
│   │   └── 类比: Linux Kernel 定义 syscalls，Framework 依赖 syscalls      │   │
│   │                                                                       │   │
│   │   Agent Framework (kbot 等):                                          │   │
│   │   ├── 定义: 基于 POSIX 接口构建的业务框架                            │   │
│   │   ├── 职责: 提供工具/记忆/编排/UI 等高层能力                        │   │
│   │   ├── 性质: 依赖 Kernel 接口，不依赖具体实现                         │   │
│   │   │                                                                 │   │
│   │   │   工具注册 / Chain 编排 / 记忆管理 / MCP 适配 / Streaming...   │   │
│   │   │                                                                 │   │
│   │   └── 类比: GNU Coreutils 调用 open/read/write，不关心 ext4 vs Btrfs│   │
│   │                                                                       │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   关键价值:                                                                  │
│   ├── 任何 Framework 可以运行在任何 Kernel 上                                │
│   ├── Kernel 实现可以替换（参考实现 → 生产实现）                            │
│   ├── 标准接口使生态系统成为可能                                            │
│   └── 安全/配额/审计 由 Kernel 保证，Framework 无法绕过                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 Agent Kernel 的核心职责

Agent Kernel 为 Agent Framework 提供 POSIX 标准接口：

| Agent Kernel 接口 | 对应 POSIX | 说明 |
|------------------|-----------|------|
| `spawn()` | `fork()` | Agent 生命周期 |
| `terminate()` | `exit()` | Agent 终止 |
| `parent()` | `getppid()` | 获取父 Agent |
| `children()` | `getchildren()` | 获取子 Agent |
| `grant()` | `chmod()` | 授予能力 |
| `verify()` | `getuid()` | 验证权限 |
| `downscope()` | `setuid()` | 降级转移 |
| `chexec()` | `execve()` | 可信执行 |
| `vault_inject()` | `keyctl()` | 凭证注入 |
| `ulimit.charge()` | `setrlimit()` | 资源计量 |
| `audit_log()` | `auditd()` | 审计记录 |
| `handoff()` | `sendmsg()` | 进程间通信 |

## 7. 为什么需要 Agent OS

### 7.1 技术驱动力

| 驱动因素 | 传统方案的问题 | Agent OS 的解决方案 |
|----------|----------------|---------------------|
| **EchoLeak 漏洞** | Prompt 注入无法防御 | chexec 污染追踪 |
| **凭证泄露** | 密钥在上下文中可见 | vault 服务端注入 |
| **权限蔓延** | 子 Agent 权限无法控制 | downscope() 强制降级 |
| **资源失控** | Token/成本不可预测 | ulimit-tok 硬限制 |
| **审计缺失** | 操作不可追溯 | audit 内容可寻址日志 |
| **碎片化** | 每个框架重复造轮子 | 统一 POSIX 接口 |
